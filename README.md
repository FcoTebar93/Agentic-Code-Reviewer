## ADMADC – Agentic Multi‑Service Dev & QA Pipeline

ADMADC is an **agentic software development pipeline**: a set of cooperating micro‑services that plan changes, generate code, review it (including security checks), and open PRs automatically while keeping a human in the loop for critical decisions.

It is designed to run **locally via Docker Compose**, while remaining flexible about which **LLM provider** and **model size** you use (mock, local 7B, or large 70B cloud models like Gemini or GPT).

---

### High‑Level Architecture

- **Frontend**
  - React dashboard to start plans, monitor events, see task status, and approve/reject PRs. It also has a token use counter and calculator, and the reasoning of each agent for their own decisions.
  - Connects to the backend via HTTP and WebSocket (through the Gateway service).
- **Gateway Service**
  - Single entry point for the frontend.
  - WebSocket `/ws`: subscribes to all events from RabbitMQ and broadcasts them to the UI.
  - HTTP proxy for:
    - `POST /api/plan` → Meta Planner.
    - `GET /api/events`, `GET /api/tasks/{plan_id}` → Memory Service.
  - Human‑in‑the‑loop (HITL) layer:
    - Holds `security.approved` events and exposes approval endpoints.
    - When a human approves/rejects, it publishes `pr.human_approved` / `pr.human_rejected`.
- **Meta Planner Service**
  - Given a high‑level user prompt and repository context, decomposes work into a **plan** and **tasks**.
  - Uses:
    - Semantic search in the Memory Service.
    - Recent events via `query_events` to avoid duplicated plans.
  - Publishes tasks to the event bus for `dev_service`.
- **Dev Service (Developer Agent)**
  - Listens for `task_assigned` events.
  - For each task:
    - Builds short‑term memory from recent events.
    - Uses tools:
      - `read_file` – preview existing code.
      - `list_project_files` – see what files exist in the target directory.
      - `run_tests` – optional automated tests after generation.
    - Calls the configured LLM to **generate or edit code**, then publishes `code_generated`.
  - Updates task state and stores events in the Memory Service.
- **QA Service (Reviewer Agent)**
  - Listens for `code_generated` events.
  - Performs a **two‑layer review**:
    - Static linting (e.g. `python_lint` for Python).
    - LLM‑based review using a structured prompt (`QA_REVIEW_PROMPT`), with:
      - Developer reasoning.
      - Short‑term memory.
      - Repository context via `search_in_repo`.
      - Language‑specific QA rules.
  - Produces:
    - `VERDICT: PASS/FAIL`.
    - Structured `ISSUES`, `REQUIRED_CHANGES`, and `OPTIONAL_IMPROVEMENTS`.
  - On PASS:
    - Marks task as `qa_passed`.
    - When all tasks for a plan pass, aggregates changes and emits `pr.requested`.
  - On FAIL:
    - Either retries with QA feedback (reassigns task to dev) or emits `qa.failed` after exhausting retries.
- **Security Service**
  - Performs additional security‑oriented scanning of code/changes.
  - Raises events like `security.blocked` or `security.approved`.
  - Approved changes still go through the **Gateway** for human approval before PRs are merged.
- **Replanner Service**
  - Listens to outcomes (e.g. `qa.failed`, `security.blocked`, `pipeline.conclusion`).
  - Uses semantic memory of outcomes to decide whether to:
    - Replan parts of the work.
    - Adjust or regenerate tasks.
  - Publishes new planning or adjustment events.
- **GitHub Service**
  - Consumes `pr.requested` events.
  - Creates branches, commits, and pull requests in the target repository.
  - Writes standardized commit messages and PR descriptions (tagged as generated by ADMADC).
  - Cooperates with the Gateway + Security Service for human approval before merging.
- **Memory Service**
  - Central event store (PostgreSQL + Qdrant for vector search, Redis for caching).
  - Exposes:
    - `/events` – query past events (by plan, type, etc.).
    - `/tasks` – store and fetch task states and code snapshots.
  - Used by all agents for short‑term memory and semantic context.
- **Shared Libraries**
  - `shared/contracts/events.py` – strongly‑typed event contracts.
  - `shared/utils` – RabbitMQ client, event bus, idempotency store, memory window builder.
  - `shared/tools` – tool registry and execution helpers.
  - `shared/llm_adapter` – unified LLM interface (providers, caching, models).

### Plan Modes (Normal vs Save)

The frontend `PlanForm` now lets you choose between two **plan generation modes**:

- **Normal mode**:
  - Default behaviour.
  - Creates and executes a full plan immediately after you submit it.
  - Good for day‑to‑day development workflows where you want the pipeline to start working right away.
- **Save mode**:
  - Generates the plan and stores it without kicking off the full execution.
  - Useful when you want to:
    - Review the generated plan first.
    - Share or iterate on the plan with your team before running it.
    - Keep a library of reusable plans for similar repositories.

The selected mode is sent as part of the request from the frontend to the planning backend, and is persisted in the Memory Service together with the plan and tasks.

### LLM Adapter & Providers

The project uses a centralized LLM adapter under `shared/llm_adapter`:

- **Factory (`get_llm_provider`)**
  - Reads `LLM_PROVIDER` (default: `mock`).
  - Returns a `CachedLLMProvider` wrapping the chosen backend.
  - Optional Redis cache (`REDIS_URL`) to deduplicate repeated prompts.
- **Supported providers (OpenAI‑compatible)**
  - `mock` – built‑in contextual mock, no external calls.
  - `openai` – OpenAI API.
  - `groq` – Groq API (supports large models like `llama-3.3-70b-versatile`).
  - `gemini` – Google AI.
  - `openrouter` – OpenRouter (meta‑provider with many models).
  - `local` – any OpenAI‑compatible local server (e.g. **Ollama**, **LM Studio**, **llama.cpp server**).
- **Configuration (from `.env` / environment variables)**
  - `LLM_PROVIDER` – selects backend (`mock`, `groq`, `local`, etc.).
  - `LLM_API_KEY` / `OPENAI_API_KEY` – API keys (ignored for most local servers).
  - `LLM_BASE_URL` – override base URL (used especially for `local`).
  - `LLM_MODEL` – override default model per provider.
  - `LLM_PROMPT_PRICE_PER_1K` / `LLM_COMPLETION_PRICE_PER_1K` – used only for **cost estimation** in dashboards.

#### Per‑service tiering

In addition to the global defaults above, each service can override its own LLM configuration via dedicated environment variables in `.env`:

- `META_PLANNER_LLM_PROVIDER`
- `DEV_LLM_PROVIDER`
- `QA_LLM_PROVIDER`
- `REPLANNER_LLM_PROVIDER`

If any of these are not set, the service falls back to `LLM_PROVIDER` and the associated defaults. This lets you, for example, run:

- A **local 7B model** for `meta_planner`, `dev_service`, and `replanner_service`.
- A **larger 70B cloud model** for `qa_service` and `security_service`.

### Recommended LLM Setup

The design intentionally supports different models **per service**, so you can balance **cost vs quality**:

- **Local “workhorse” model (7B)**
  - Use for:
    - `dev_service` (code generation and refactors).
    - `meta_planner`, `replanner`, and other high‑frequency tasks.
  - Typical choice:
    - **Qwen 2.5 Coder 7B** or **Llama 3.x 8B** via **Ollama**:
      - `LLM_PROVIDER=local`
      - `LLM_BASE_URL=http://host.docker.internal:11434/v1`
      - `LLM_MODEL=qwen2.5-coder:7b` (or the exact model tag in Ollama).
- **Large “expert” model (≈70B)**
  - Use for:
    - `qa_service` – deep code review and quality gate.
    - `security_service` – security‑oriented analysis and approval.
  - Typical choice:
    - **LLaMA 3.3 70B** (or similar) via **Groq**:
      - `LLM_PROVIDER=groq`
      - `LLM_MODEL=llama-3.3-70b-versatile` (or your chosen 70B model).
      - `LLM_API_KEY=<your_groq_key>`.

When using per‑service tiering, keep `LLM_PROVIDER` and `LLM_MODEL` set to a **safe default** (e.g. `mock` or a local 7B) and configure `QA_LLM_PROVIDER` and `QA_LLM_MODEL` only in environments where you have access to the larger cloud model.

In this configuration, the 7B local model generates and adjusts code, while the 70B model acts as a strict senior reviewer for QA and security. The QA prompts are structured so the 70B returns clear `ISSUES` and `REQUIRED_CHANGES` that the 7B simply has to follow.

### Tools & Static Analysis

Tools are orchestrated by the services (the LLM does not directly choose tools yet):

- **Meta Planner**
  - `semantic_search_memory` – semantic search in Memory Service (Qdrant).
  - `query_events` – recent events for temporal context and duplicate‑plan avoidance.
- **Dev Service**
  - `read_file` – grab existing file contents as context.
  - `list_project_files` – list files in the target directory (per language glob).
  - `run_tests` – run project tests (configurable commands per language).
- **QA Service (by default)**
  - `python_lint` – Python static analysis (ruff, via pip).
  - `python_security_scan` – Python security analysis (Bandit, via pip).
  - `semgrep_scan` – multi‑language static/security analysis (Semgrep, via pip).
  - `search_in_repo` – quick search for related usages in the repo.
  - `query_events` – build a short‑term memory window of relevant events.
- **QA Service (optional extra linters)**
  - JS/TS linting with ESLint.
  - Java compile‑time checks with `javac`.
  - These are disabled by default; see the section below on enabling them.
- **Replanner Service**
  - `semantic_outcome_memory` – semantic memory of past outcomes (qa.failed, security.blocked, pipeline.conclusion).
- **Security Service**
  - Uses its own internal scanning logic (regex/patterns) rather than the shared ToolRegistry.

### Running the Project

1. **Clone the repository**

```bash
git clone https://github.com/FcoTebar93/Agentic-Project.git
cd Agentic-Project
```

1. **Create `.env`**

- Copy `.env.example` to `.env` and adjust values:
  - PostgreSQL, RabbitMQ, Redis passwords.
  - `RABBITMQ_URL`, `DATABASE_URL`, `QDRANT_URL`, `REDIS_URL`.
  - LLM config:
    - Start with `LLM_PROVIDER=mock` for a zero‑dependency local setup.
    - Or configure `LLM_PROVIDER=local` / `groq` as described above.

1. **Start the stack with Docker Compose**

```bash
docker compose up --build
```

- This should bring up:
  - Services (gateway, meta_planner, dev, qa, security, replanner, memory, github).
  - Database, Redis, RabbitMQ, Qdrant.
  - Frontend dashboard.

1. **Access the UI**

- Open the frontend in your browser (see `docker-compose.yml` for the exposed port).
- From the UI you can:
  - Create a new plan (enter project name, repo URL, and prompt).
  - Watch tasks flow through dev → QA → security.
  - Approve/reject PRs from the human‑in‑the‑loop view.

### Enabling optional JS/TS & Java linters in QA

By default, the QA service relies on **Python tools (ruff, Bandit)** and **Semgrep** for multi‑language static analysis.  
If you also want **ESLint (JS/TS)** and **javac (Java)** checks to run inside the `qa_service` container:

- Open `infrastructure/docker/qa_service/Dockerfile`.
- Locate the commented block:

```dockerfile
# Optional: enable JS/TS and Java linters inside the qa_service container.
# RUN apt-get update && \
#     apt-get install -y --no-install-recommends nodejs npm openjdk-17-jdk && \
#     npm install -g eslint typescript && \
#     apt-get clean && rm -rf /var/lib/apt/lists/*
```

- **Uncomment** the `RUN` lines, then rebuild:

```bash
docker compose build qa_service
docker compose up
```

With this enabled:

- `js_ts_lint` uses ESLint to analyse JavaScript/TypeScript code.
- `java_lint` uses `javac` to surface basic compile‑time errors in Java code.
- `semgrep_scan` continues to run as a multi‑language static/security analyzer.

### Development Workflow

- **Backend services**
  - Written in Python (FastAPI).
  - Each service has its own `main.py` and `config.py`.
  - Contracts and shared logic live under `shared/`.
- **Frontend**
  - React app under `frontend/`.
  - Connects to `gateway_service` HTTP and WebSocket endpoints.
- **Observability**
  - Metrics endpoints (`/metrics`) per service.
  - Structured JSON logging via `shared/logging/logger.py`.

---

### Current Limitations

- Lints and tools are richer for **Python** than for other languages (e.g. `python_lint` via ruff).
- The LLM currently does **not** perform true “tool use” (function calling) by itself:
  - The orchestrator services decide when to call tools and how to inject their outputs into prompts.
- Security scanning is primarily static and pattern‑based; there is no full SAST/DAST integration yet.
- Long, cross‑service refactors still rely heavily on the quality of the meta‑planner prompt and the underlying model.

### Roadmap / Future Work

- **Deeper LLM tiering**
  - Formalize per‑service LLM configuration:
    - 7B local for dev/meta/replanner.
    - 70B cloud for QA and security.
  - Optional fallback to a larger model when the local model is uncertain or when risk is high.
- **Richer tooling per language**
  - Add lint and test tools for:
    - JavaScript / TypeScript (ESLint, TypeScript compiler).
    - Java (javac, static analysis tools).
  - Integrate security‑focused tools (Bandit, Semgrep, etc.) into QA and Security services.
- **True tool‑use by LLMs**
  - Expose `read_file`, `list_project_files`, `search_in_repo`, `query_events`, etc. as function‑calling interfaces.
  - Let certain agents (especially dev and QA) decide autonomously when to pull more context.
- **Improved memory and context management**
  - Smarter short‑term memory windows tuned to each service.
  - Better summarization of long event histories to keep prompts compact.
  - Cross‑plan semantic memory for learning from past failures and successes.
- **More robust security pipeline**
  - Dedicated `security_service` enhancements:
    - Deeper analysis of auth flows and data handling.
    - Stronger detection of dangerous patterns per framework.
  - Multi‑stage approvals: QA, security, and human approvals for high‑risk changes.
- **Developer experience**
  - Richer frontend for:
    - Visualizing plan graphs, dependencies, and task state transitions.
    - Inspecting LLM reasoning (dev + QA + security) in a unified timeline.
  - CLI tooling to trigger runs and inspect plans from the terminal.
